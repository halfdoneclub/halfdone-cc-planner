# PRD 작성 가이드 (지식 파일)

> Tier 2, Tier 3에서 PRD를 작성할 때 참조하는 가이드.

---

## PRD 필수 섹션

```
# 프로젝트명

## 1. 개요
- 한 줄 요약
- 배경과 목적
- 성공 기준: 이 프로젝트가 "완성"이라고 할 수 있는 조건

## 2. 사용자 및 시나리오
- 주요 사용자
- 핵심 사용 시나리오 (2~3개)

## 3. 핵심 기능
- 기능별 설명
- 우선순위: P0 (필수) / P1 (중요) / P2 (있으면 좋음)

## 4. 기술 요구사항
- 기술 스택
- 성능/환경 기준
- 주요 제약

## 5. 범위 외 항목
- 이번에 하지 않는 것을 명시

## 6. Claude Code 구현 가이드
- 구현 순서 (P0부터)
- 기능별 완료 확인 방법
```

---

## 프로젝트 유형별 추가 섹션

위 필수 섹션 외에, 프로젝트 유형에 따라 아래를 추가한다:

| 프로젝트 유형 | 추가 섹션 | 포함 내용 |
|-------------|----------|----------|
| **웹/앱** | 화면 구조 | 화면 목록, 화면 간 이동 흐름, 반응형 기준 |
| **CLI 도구** | 명령어 설계 | 명령어 목록, 옵션/플래그, 입출력 형식 |
| **API/백엔드** | API 설계 | 엔드포인트 목록, 데이터 모델, 인증 방식 |
| **데이터 처리** | 파이프라인 설계 | 입출력 스키마, 처리 단계, 에러 핸들링 |

## PRD 작성 원칙

1. **성공 기준을 반드시 포함한다.** "이 프로젝트가 끝났다"고 할 수 있는 조건이 없으면 구현 시 범위가 계속 늘어난다.

2. **범위 외 항목을 명시한다.** "이번에 하지 않는 것"을 적어야 불필요한 기능 추가를 막는다.

3. **기능 우선순위를 매긴다.** P0/P1/P2로 나누면 Claude Code에서 구현 순서가 명확해진다. P0만 먼저 구현하고 확인한 뒤 P1으로 넘어간다.

4. **기술 스택이 없으면 비워둔다.** 사용자가 선호가 없으면 Claude Code가 적합한 스택을 선택하게 한다. 억지로 정하지 않는다.

5. **추상적 기능 서술을 피한다.** "사용자 친화적 인터페이스"가 아니라 "검색 필드 1개 + 결과 리스트 + 상세 페이지"처럼 구체적으로 쓴다.

---

## Claude Code에 PRD를 전달하는 방법

1. PRD 파일을 프로젝트 폴더에 저장한다: `/docs/PRD.md`
2. Claude Code에서 다음과 같이 시작한다:

```
/docs/PRD.md를 읽고 이 기반으로 구현해줘.
P0 기능부터 시작하고, 각 기능이 끝날 때마다 확인시켜줘.
한 번에 전부 만들지 말고, 기능 단위로 나눠서 진행해줘.
```

3. 각 기능이 끝날 때마다 확인하고, 문제가 있으면 바로 수정한다.
4. P0 완료 → P1 → P2 순서로 진행한다.
